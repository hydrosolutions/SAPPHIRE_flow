# REST API and Dashboard

## API design principles

- API-first: the dashboard is a client of the API, not the other way around
- All forecast data, station data, and management operations accessible via REST
- OpenAPI/Swagger docs auto-generated by FastAPI
- Versioned: `/api/v1/...`
- JSON responses throughout

### Pagination

All list endpoints use cursor-based pagination (not offset-based, which degrades
on large tables):

```
GET /api/v1/stations?limit=50&after=<cursor>
```

- `limit`: max items per page (default 50, max 200)
- `after`: opaque cursor from previous response's `next_cursor` field
- Response includes `next_cursor` (null if no more pages) and `total_count`
- Time-series endpoints (observations, forecasts) enforce a maximum query range
  of 1 year per request. For larger ranges, use the `/export/training-data`
  streaming endpoint.

## Authentication and security

- **fastapi-users** library backed by PostgreSQL
- Cookie-based sessions for dashboard users:
  - `Secure; HttpOnly; SameSite=Strict` flags enforced
  - Idle timeout: 30 minutes. Absolute timeout: 12 hours.

### Session management

- The idle timeout (30 min) resets on every authenticated request — active use
  keeps the session alive. HTMX partial page updates count as requests.
- A client-side JavaScript timer warns the user at 25 minutes of inactivity
  with an option to extend the session (triggers a lightweight keepalive request).
- If a session expires during a multi-step workflow (e.g. bulletin review), the
  user is redirected to login. Previously selected forecast IDs are preserved
  in the URL or a session-recovery cookie so work is not lost on re-authentication.

### CSRF protection

All state-changing requests (POST, PATCH, PUT, DELETE) from the dashboard require
a CSRF token. Implementation:

- The server generates a per-session CSRF token and embeds it in a `<meta>` tag in the base Jinja2 template (`<meta name="csrf-token" content="...">`). The token is also stored server-side in the session.
- HTMX requests include the token via a custom header (`X-CSRF-Token`) configured globally: `<body hx-headers='{"X-CSRF-Token": "..."}'>` (value read from the meta tag)
- The server validates the header against the session-stored token on every mutating request
- `SameSite=Strict` on session cookies provides defense-in-depth but is not sufficient alone (does not protect against subdomain attacks or certain browser plugin vulnerabilities)
- API consumers using JWT or scoped bearer tokens are exempt (token-based auth is inherently CSRF-resistant)

- TOTP-based MFA required for `forecaster` and `admin` roles on the dashboard
- Short-lived JWT (15 min) with refresh token rotation for API consumers

### JWT refresh token lifecycle

- Refresh tokens are stored as hashed values in the database (via fastapi-users session backend)
- Lifetime: 7 days
- **Rotation**: each use of a refresh token issues a new access + refresh token pair and invalidates the old refresh token
- **Reuse detection**: if a previously-used (invalidated) refresh token is presented, ALL sessions for that user are revoked (indicates token theft)
- **Logout**: deletes the refresh token from the database
- Refresh tokens are bound to the originating IP and user agent — significant changes require re-authentication

- Scoped bearer tokens for external institutions (mandatory expiry, max 12 months)
- Rate limiting via Caddy reverse proxy (see 07-deployment.md)
- All traffic over TLS — Caddy handles certificate management
- Roles: `viewer`, `forecaster`, `admin`

| Role        | Can view data | Can adjust forecasts | Can manage users/config |
|-------------|---------------|----------------------|------------------------|
| viewer      | yes           | no                   | no                     |
| forecaster  | yes           | yes                  | no                     |
| admin       | yes           | yes                  | yes                    |

### Password and brute-force policy

- Minimum 12 characters, at least one uppercase, one lowercase, one digit
- Passwords checked against a common-passwords list (top 10,000) at registration
- Account lockout after 5 failed login attempts within 15 minutes (30-minute lockout)
- Lockout events logged to `audit_log` with client IP
- Rate limiting on `/auth/login` via Caddy (stricter than general API rate limit)
- Password reset requires admin action (no self-service email reset in v1.0 — operational hydromets have small user counts)
- Admin-initiated password resets are logged to `audit_log` with `action = 'password_reset'` and the target user ID in `detail`
- The affected user receives a notification (email if configured) that their password was reset by an administrator

### Break-glass account recovery

- **Locked-out user**: A server operator can run `docker compose exec api sapphire-flow unlock-user --username <user>` to immediately clear a lockout. Logged to `audit_log`.
- **Lost TOTP device**: Admin can reset a user's MFA via `docker compose exec api sapphire-flow reset-mfa --username <user>`. The user must re-enroll TOTP on next login.
- **Locked-out admin with lost TOTP**: Server operator runs both `unlock-user` and `reset-mfa` via CLI. This is the only recovery path and requires SSH access to the server.
- **Backup codes**: At TOTP enrollment, 10 one-time backup codes are generated and displayed once. These can be used in place of TOTP for account recovery.

### Scoped access tokens

Admins can create scoped tokens for external institutions:

```
POST /api/v1/admin/tokens
{
  "name": "Ministry of Water - Region A",
  "scope": {
    "stations": ["ABC-001", "ABC-002", "ABC-003"],
    "forecast_status": ["published"],
    "read_only": true
  },
  "expires_at": "2027-12-31T00:00:00Z"
}
```

Expiry is mandatory — maximum 12 months. Tokens must be renewed before expiry.

Requests bearing a scoped token are filtered server-side — the consumer
only sees data matching their scope. No client-side enforcement.

Null semantics: `scope.stations = null` grants access to all stations;
`scope.stations = []` grants access to none. See 02-data-model.md for the
full specification.

## API endpoints

### Stations

```
GET    /api/v1/stations                      List all stations (filterable by basin)
GET    /api/v1/stations/{id}                 Station details
GET    /api/v1/stations/{id}/observations    Time series data
GET    /api/v1/stations/compare              Multi-station comparison (see below)
GET    /api/v1/stations/{id}/observations?format=csv    Download as CSV
GET    /api/v1/stations/{id}/forecasts?format=csv       Download as CSV
```

The `format=csv` query parameter returns a CSV file instead of JSON. Useful
for downstream tools (Excel, HEC-RAS, custom scripts).

**CSV injection protection**: All string values in CSV exports are sanitized.
Cells starting with `=`, `+`, `-`, `@`, `\t`, or `\r` are prefixed with a
single quote (`'`) to prevent formula execution when opened in Excel. This
applies to station names, parameter names, free-text fields (alert notes,
adjustment reasons), and any user-supplied content.

### Station management

```
GET    /api/v1/stations                          List all stations (filterable by basin, kind)
GET    /api/v1/stations/{id}                     Station details + metadata
POST   /api/v1/stations                          Create station (admin only)
PATCH  /api/v1/stations/{id}                     Update station metadata (admin only)
POST   /api/v1/stations/virtual                  Create virtual station (admin only)
```

Station metadata is managed in the database, not TOML config. Virtual stations
(ungauged forecast sites with derived runoff) are created via the `/virtual`
endpoint and require a `derivation_config` field specifying how runoff is
calculated (e.g. upstream station references, area ratios).

### Observations

```
GET    /api/v1/stations/{id}/observations    Time series with quality flags
PATCH  /api/v1/observations/{station}/{param}/{timestamp}  Edit an observation
POST   /api/v1/observations/{station}/{param}/{timestamp}/exclude  Flag as "do not use"
POST   /api/v1/observations/{station}/{param}/{timestamp}/restore  Undo exclusion
POST   /api/v1/observations/batch-edit         Batch correct/exclude observations
```

Editing and excluding require `forecaster` or `admin` role. Every edit
creates an immutable `observation_edits` record.

`batch-edit` accepts a station, parameter, time range, and either an offset
(e.g. +5.0 cm) or an exclusion flag. Creates individual `observation_edits`
records for each affected observation. Essential for correcting known sensor
offsets across a time period.

### Forecasts

```
GET    /api/v1/forecasts/latest              Latest forecast per station
GET    /api/v1/forecasts/{id}                Specific forecast with ensemble
GET    /api/v1/stations/{id}/forecasts       Forecast history for a station
GET    /api/v1/forecasts/compare             Multi-station forecast comparison
GET    /api/v1/stations/{id}/forecasts/models  All models' latest forecasts for this station

POST   /api/v1/forecasts/{id}/adjust         Submit a manual adjustment
GET    /api/v1/forecasts/{id}/adjustments    Adjustment audit log
PATCH  /api/v1/forecasts/{id}/status         Set status (reviewed/selected/published)
```

The `/models` endpoint returns the latest forecast from each model for a
single station, with ensemble summaries (median, 10th/90th percentile).
Designed for side-by-side model comparison on the same chart — the
hydrologist sees LSTM vs HBV vs persistence on the same axes.

#### Aggregation query parameter

Daily forecasts can be aggregated on the fly:

```
GET /api/v1/stations/{id}/forecasts?aggregate=pentadal
GET /api/v1/stations/{id}/forecasts?aggregate=dekadal
```

This computes 5-day or 10-day means from daily forecast values. If a model
natively produces pentadal/seasonal output, those are stored and returned
directly (no aggregation needed).

Valid values: `pentadal` (5-day), `dekadal` (10-day), `monthly`.

Aggregation respects the parameter's `aggregation_method`: precipitation is summed,
temperature is averaged, water level and discharge use the mean. The method is
defined per parameter in the database (see 02-data-model.md) and applied
automatically — callers do not specify the aggregation method.

Pentadal aggregation uses calendar pentads (days 1-5, 6-10, 11-15, 16-20,
21-25, 26-end of month), not rolling 5-day windows. Dekadal uses days 1-10,
11-20, 21-end.

#### Multi-station comparison

```
GET /api/v1/forecasts/compare?ids=ABC-001,DEF-002,GHI-003&parameter=discharge
GET /api/v1/stations/compare?ids=ABC-001,DEF-002,GHI-003&parameter=water_level&start=2026-11-01&end=2026-11-15
```

Returns aligned time series for up to 6 stations, suitable for overlay
charts. Works for both observations and forecasts.

### Flood alerts

```
GET    /api/v1/alerts                        Active alerts (unresolved), sorted by severity
GET    /api/v1/alerts?level=danger           Filter by severity
GET    /api/v1/alerts/history                Past resolved alerts (for post-event review)
POST   /api/v1/alerts/{id}/acknowledge       Mark alert as seen by forecaster
POST   /api/v1/alerts/{id}/notes             Add operational notes to an alert
GET    /api/v1/stations/{id}/thresholds      Flood thresholds for a station
PUT    /api/v1/stations/{id}/thresholds      Update thresholds (admin only)

GET    /api/v1/alerts?source=forecast      Filter by alert source
GET    /api/v1/alerts?source=observation   Observation-triggered alerts only
```

Alerts are persisted in the `alert_events` table with a lifecycle: raised -> acknowledged -> resolved. Alerts can be triggered by forecasts (when predicted values exceed thresholds) or by real-time observations (when measured values exceed thresholds). The `source` query parameter filters by trigger type. New forecast-based alerts are created when a forecast exceeds thresholds; observation-based alerts are created when an ingested observation exceeds thresholds. Both types are auto-resolved when newer data no longer exceeds them. The priority inbox shows unacknowledged alerts first, then acknowledged-but-unresolved, sorted by severity.

### Rating curves

```
GET    /api/v1/stations/{id}/rating-curves         All versions
GET    /api/v1/stations/{id}/rating-curves/active   Current active curve
POST   /api/v1/stations/{id}/rating-curves          Upload new curve
```

### Bulletins

```
GET    /api/v1/bulletins                     List generated bulletins
GET    /api/v1/bulletins/{id}                Bulletin metadata
GET    /api/v1/bulletins/{id}/download       Download generated Excel file
POST   /api/v1/bulletins/generate            Trigger bulletin generation
POST   /api/v1/bulletins/preview             Preview bulletin data (JSON) without generating file
```

`/preview` returns the data that would populate the template, without
actually generating the Excel file. Allows the hydrologist to verify
contents before committing.

Bulletin generation request:

```json
{
  "scope": "country",
  "template_id": "daily_forecast_v2",
  "forecast_ids": ["uuid1", "uuid2", "..."]
}
```

Or per-basin:

```json
{
  "scope": "basin",
  "basin_id": "uuid",
  "template_id": "basin_report_v1",
  "forecast_ids": ["uuid1", "uuid2"]
}
```

The backend uses `ieasyreports` to fill the Excel template with the
selected (and possibly adjusted) forecast data, then stores the result.

### Model skill

```
GET    /api/v1/stations/{id}/skill           Verification scores for all models at this station
GET    /api/v1/stations/{id}/skill?model=lstm_daily  Filter by model
GET    /api/v1/skill/overview                Summary table: all stations x models
```

Returns metrics (NSE, CRPS, etc.) by lead time, allowing the hydrologist
to assess which model performs best and how skill degrades with lead time.

### Data export (for model training)

```
GET    /api/v1/export/training-data              Bulk export for model training
       ?station=ABC-001                           Required: station code
       &start=2000-01-01&end=2025-12-31          Optional: time range (default: all)
       &format=csv|parquet                        Optional: format (default: csv)
       &include=observations,weather,metadata     Optional: what to include (default: all)
```

Returns a streamed export of:
- Observations (with quality flags, excluding flag=9)
- Weather forecast history (all ensemble members)
- Station metadata (location, elevation, basin area)
- Rating curve versions (if applicable)

Requires `admin` or `forecaster` role. Designed for annual or multi-year
retraining workflows. Large exports are streamed to avoid memory pressure.

### Operations

```
POST   /api/v1/flows/ingest/trigger          Trigger ingest manually
POST   /api/v1/flows/forecast/trigger         Trigger forecast for all stations
POST   /api/v1/flows/forecast/trigger?station=ABC-001   Single station
POST   /api/v1/flows/forecast/trigger?basin=BASIN-01    All stations in a basin
GET    /api/v1/health                         System health + last run times
```

**Rate limiting**: Flow trigger endpoints are rate-limited to 5 requests per
minute per user at the application level (FastAPI middleware). This prevents
accidental or malicious repeated triggering from overwhelming the worker and
database. This is in addition to Caddy's general rate limit.

```
GET    /api/v1/operations/summary             Morning briefing: last run status, stale stations, active alerts
```

`/operations/summary` returns a single JSON object summarizing system state:
last ingest times (weather + stations), last forecast time, number of
stations with stale data, count of active alerts by severity, any failed
flows in the last 24 hours. Designed for the morning shift handover.

### Admin

```
POST   /api/v1/admin/tokens                  Create scoped access token
GET    /api/v1/admin/tokens                  List active tokens
DELETE /api/v1/admin/tokens/{id}             Revoke a token
```

## Forecast adjustment workflow

1. Forecaster views the latest forecast on the dashboard (or via API)
2. Forecast shows ensemble spread: median, 10th/90th percentile bands
3. Forecaster can adjust individual lead times or shift the entire forecast
4. On submit:
   - Original values are snapshot to `forecast_adjustments` table
   - Adjusted values are stored
   - Forecaster's user ID + timestamp + optional reason are logged
5. Forecaster sets status to `reviewed`
6. Downstream consumers (API, bulletins) see the adjusted forecast
7. Audit trail is immutable — no edits, only new adjustment records

### Concurrency control

Forecast status transitions use optimistic concurrency via a `version` column:

- Every status change request includes the expected `version` (from the last GET)
- `UPDATE forecasts SET status = ?, version = version + 1 WHERE id = ? AND version = ?`
- If the version does not match (another user changed it), the server returns
  HTTP 409 Conflict with the current state
- Valid transitions are enforced server-side: `raw → reviewed → selected → published`.
  Invalid transitions (e.g. `raw → published`) return HTTP 422.
- For adjustments: concurrent edits to the same lead time are last-writer-wins
  (each creates an immutable audit record regardless). The dashboard polls for
  updates and warns if the forecast was modified by another user.

### Published forecast immutability

Forecasts with `status = published` are frozen — adjustments are rejected by the
API with HTTP 409 and the message "Cannot adjust a published forecast." If a
correction is needed post-publication, the forecaster must:

1. Create a new bulletin with corrected forecasts
2. The old bulletin is flagged as `superseded` in the bulletin record
3. Both bulletins remain in the audit trail

## Bulletin workflow

1. **Forecast tab**: Hydrologist reviews forecasts from different models for each station.
   Selects the preferred forecast (from a specific model) per station.
   Sets status to `selected`.

2. **Bulletin tab**: Hydrologist sees all selected forecasts.
   Can make final edits (adjustments) specific to the bulletin.
   Chooses a template from the available Excel templates.

3. **Generate**: Triggers bulletin generation:
   - Country-level: all selected forecasts across the deployment
   - Per-basin: selected forecasts for stations in that basin
   - Uses `ieasyreports` library to fill the Excel template
   - Generated file is stored and downloadable via API

4. **Publish**: Generated bulletin can be shared. Forecast statuses
   are updated to `published`.

## Observation editing workflow

1. Hydrologist views station data and spots a suspicious value
2. Two options:
   - **Correct**: Replace the value with a corrected one + reason
   - **Exclude**: Flag as "do not use for forecasting" + reason
3. Both create an immutable `observation_edits` audit record
4. Excluded observations are filtered out of forecast model inputs
5. Corrected values replace the original in-place (original preserved in audit log)

## Flood alert and notification workflow

1. After each forecast run, the system checks median forecast values
   against flood thresholds for each station
2. Exceedances create `alert_events` records (persisted, not ephemeral)
3. Dashboard shows a **priority inbox** — unacknowledged alerts first,
   sorted by severity (danger > warning > watch)
4. Forecaster acknowledges alert ("I've seen this") and optionally adds notes
5. When a newer forecast no longer exceeds the threshold, the alert is auto-resolved
6. Resolved alerts remain in history for post-event review and accountability
7. Notifications:
   - **Mandatory** for danger-level alerts (at least email or webhook)
   - Configurable for warning-level alerts
   - Uses a pluggable NotificationSink (not tied to Prefect)

## Dashboard

The dashboard is a set of server-rendered HTML pages served by FastAPI,
using HTMX for interactivity. It is optional — hydromets can use the
API directly with their own UI.

### Key views

1. **Country overview**: Map view with station dots colored by status
   (green = fresh data + no alerts, yellow = stale data, orange = watch/warning,
   red = danger). Table below with sortable columns. Includes a trend
   indicator (rising/stable/falling) per station based on recent observations.

2. **Flood priority inbox**: Filtered view showing only stations with
   active alerts, sorted by severity. One-click navigation to station detail.

3. **Station detail**: Time series chart showing recent observations +
   forecast ensemble (median + uncertainty bands). Observation editing
   controls. Interactive forecast adjustment.

4. **Multi-station comparison**: Select up to 6 stations, overlay their
   observations or forecasts on the same chart. Useful for upstream/downstream
   analysis.

5. **Forecast tab**: Per-station view with model comparison chart: overlay
   latest forecast from each model (median + uncertainty bands) on the same
   axes, alongside recent observations. Hydrologist selects preferred model,
   reviews skill scores, sets forecast status.

6. **Model skill**: Historical performance charts per station and model.
   Skill vs lead time plots. Reliability diagrams.

7. **Bulletin tab**: Selected forecasts ready for bulletin production.
   Final adjustment controls. Template selection. Generate button
   (country-wide + per-basin). Download generated files.

8. **Rating curves**: View current curve, upload new version, see history.

9. **Run status**: Link to Prefect UI or embedded summary of recent
   flow runs (success/failure/pending). Manual re-trigger buttons
   (per station, per basin, all).

10. **Operations summary**: Morning briefing view showing last run status
    for all flows, stations with stale data, and a quick alert count.
    Embedded — no need to visit the Prefect UI.

### Localization

v1.0 supports Nepal-specific localization:
- Date display in Bikram Sambat (Nepali calendar) alongside Gregorian
- Times displayed in Nepal Standard Time (UTC+5:45)
- Interface language: English (Nepali language support deferred)

v3.0 will add:
- Central Asian time zones
- Russian and local language support (Kyrgyz, Tajik, etc.)
- Configurable calendar systems per deployment

### Technology

- **HTMX**: HTML-over-the-wire — server renders HTML fragments, HTMX swaps
  them into the page. No JavaScript framework, no build step.
- **CSS framework**: Pico CSS or Simple.css — lightweight, classless or minimal-class.
- **Charts**: A lightweight JS charting library for time series plots
  (e.g. Chart.js, Plotly.js, or uPlot for performance).
- **Maps**: Leaflet.js — lightweight, no API key needed, works offline with
  tile caching.
- **Templates**: Jinja2 (built into FastAPI/Starlette).

### Why not a JS frontend?

- One language (Python) for the entire codebase
- No Node.js, npm, webpack/vite to install or maintain
- Hydrologists can read and modify templates
- HTMX provides sufficient interactivity for this use case
- Charts and maps are the only parts that need JS — single library imports

### When a hydromet should use their own UI

If a hydromet:
- Already has a web platform for operational data
- Has frontend developers who prefer React/Vue/etc.
- Wants to integrate forecasts into their existing dashboard

They skip the HTMX dashboard entirely and consume the REST API.
The API is the product; the dashboard is a convenience.
